**1. Concepto de Proceso y Diferencias Fundamentales**
El concepto central de cualquier sistema operativo (SO) es el proceso, definido 
informalmente como un programa en ejecución [1]. 
La diferencia primordial radica en que **un programa es una entidad pasiva** (un
archivo almacenado en disco con instrucciones), mientras que **un proceso es una
entidad activa** que posee un contexto, recursos asociados y un estado de 
ejecución [2].
En sistemas multiprogramados, el procesador cambia rápidamente entre procesos, 
creando una ilusión de paralelismo (pseudoparalelismo), a diferencia del 
verdadero paralelismo de hardware que se da cuando hay múltiples procesadores 
físicos [1, 3].

**2. Imagen del Proceso en Memoria**
Cuando un proceso se carga en memoria, su representación física se divide en las
siguientes secciones [2, 4, 5]:
*   **Contexto:** Contiene los registros utilizados por el SO para gestionarlo, 
como el PC (Program Counter), IR (Instruction Record) y PSW (Program Status 
Word).
*   **Texto (Código):** Contiene las instrucciones que conforman el programa.
*   **Datos:** Almacena datos y variables globales.
*   **Pila (Stack):** Almacena datos temporales como variables locales, 
parámetros de funciones, y direcciones de retorno. Su tamaño es dinámico y 
típicamente crece hacia abajo.
*   **Heap (Montículo):** Se usa para la asignación dinámica de memoria (ej. 
listas, colas). Crece hacia arriba.

**3. Estructuras de Datos del SO y el BCP**
El SO administra el sistema apoyándose en cuatro tipos de tablas: Tablas de 
Memoria, Tablas de Dispositivos (E/S), Tablas de Archivos y Tablas de Procesos 
[6]. 
La Tabla de Procesos tiene una entrada por cada proceso del sistema, constituida
por el **Bloque de Control de Proceso (BCP o PCB)**. El BCP es la estructura de 
datos clave que permite la multiprogramación, e incluye [7-9]:
*   **PID:** Identificador único.
*   **Estado y Prioridad.**
*   **Datos de contexto:** Valores de los registros (PC, IR) que se guardan al 
pausar un proceso para luego poder restaurarlo [10].
*   **Información adicional:** Peticiones de E/S pendientes, punteros a memoria 
de código/datos, y datos de auditoría (tiempo de uso de procesador) [8].

**4. Modelos de Estados y Transiciones**
El SO controla la ejecución de los procesos haciéndolos transitar por diferentes
estados.

**Modelo de 2 Estados:**
Un proceso solo puede estar **Ejecutando** o **No Ejecutando** [11]. Los 
procesos detenidos esperan en una cola FIFO (Primero en Entrar, Primero en 
Salir) gestionados por un pequeño programa llamado *dispatcher* (despachador), 
utilizando una estrategia de planificación (scheduling) cíclica o *Round-Robin* 
[12-14].

**Modelo de 5 Estados:**
Para mejorar la eficiencia, el estado "No Ejecutando" se desdobla [14, 15]:
*   **Nuevo:** El proceso se creó (tiene BCP y PID), pero su código aún no se ha
cargado en la memoria principal (permanece en almacenamiento secundario) [16, 
17].
*   **Listo:** Cargado en memoria y esperando turno de procesador [15].
*   **Ejecutando:** Usando activamente el procesador [15].
*   **Bloqueado:** Pausado obligatoriamente esperando que ocurra un evento o 
finalice una operación de E/S [15, 16].
*   **Saliente:** El proceso finalizó o fue abortado [16].
*   *Transiciones clave:* De Ejecutando a Listo (agotó su tiempo o por 
prioridad), de Listo/Bloqueado a Saliente (si su proceso "padre" lo finaliza), y
de Bloqueado a Listo (cuando ocurre el evento esperado) [18, 19].

**Modelo de 7 Estados (Con Swapping):**
Si la memoria RAM se llena de procesos Bloqueados, el procesador queda ocioso. 
Para resolverlo, el SO usa **Swapping** (memoria de intercambio, considerada una
operación de E/S), que mueve parcial o totalmente un proceso de la memoria 
principal al disco [20-22]. Se suman dos estados [21, 23]:
*   **Listo / Suspendido:** El proceso está en disco, pero listo para ejecutarse
si se trae a RAM [23].
*   **Bloqueado / Suspendido:** El proceso está en disco y además esperando un 
evento [23].

**5. Control de Procesos y Modos de Ejecución**
El procesador posee un bit en el registro PSW (Palabra de Estado del Programa) 
que define su modo de ejecución [6, 24]:
*   **Modo Usuario:** Restringido, existe para proteger al sistema. La memoria 
se accede de forma relativa y solo puede solicitar servicios al SO mediante 
*Syscalls* [6].
*   **Modo Núcleo (Kernel):** Control absoluto del sistema y referencias físicas
a memoria. Aquí se ejecutan tareas críticas como creación de procesos, 
planificación (scheduling), swapping y gestión de interrupciones [6, 25].

**6. Cambio de Contexto y Excepciones**
Un **cambio de contexto (Context Switch)** ocurre cuando el SO toma el control, 
interrumpe el proceso actual, salva su contexto en su BCP, y restaura el 
contexto de otro proceso [25, 26]. Es provocado por [26-35]:
*   **Interrupciones:** Eventos asíncronos del hardware (ej. reloj que agota el 
tiempo del proceso, dispositivos de E/S que terminan su tarea) [27, 28].
*   **Llamadas al Sistema (Syscalls):** Petición voluntaria del programa (ej. 
leer un archivo o crear un proceso) [34, 35].
*   **Excepciones:** Errores síncronos internos del proceso [29, 30]. Se 
clasifican en:
    *   **Abort:** Errores fatales de hardware o tablas corruptas. El proceso 
pasa a estado Saliente [31].
    *   **Fault:** Errores corregibles. El ejemplo clásico es un *Fallo de 
página* (acceder a memoria que está en disco). El proceso se bloquea, el SO trae
la página a RAM, y la instrucción que falló se reinicia [31, 32].
    *   **Trap:** Se informa justo después de ejecutar la instrucción y permite 
continuar (ej. punto de interrupción/breakpoint para debug o división por cero) 
[33, 34].

*Nota: Un **Cambio de Modo** (pasar de Usuario a Kernel) ocurre para atender 
estos eventos. Sin embargo, no siempre implica un Cambio de Contexto completo 
si, al terminar la rutina del SO, se reanuda el mismo proceso de usuario que fue
interrumpido* [35-37].

**7. Ejecución del propio Sistema Operativo**
Siendo software, el SO también debe ser ejecutado por el procesador. Hay tres 
enfoques de diseño [35, 38]:
1.  **Núcleo sin procesos:** El SO es un software monolítico que corre 
totalmente aislado. Cada interrupción genera un cambio de contexto completo [35,
39].
2.  **Dentro de procesos de usuario:** Las rutinas del SO se ejecutan 
virtualmente dentro del entorno (y pila) del proceso de usuario actual. Es 
eficiente porque evita dobles cambios de contexto (el proceso actúa como un 
"contenedor") [37, 40-42].
3.  **Basado en procesos:** El SO es una colección de procesos independientes 
ejecutándose en modo núcleo. Impone gran modularidad y su mayor ventaja es 
permitir el paralelismo (múltiples procesos del sistema ejecutándose 
simultáneamente) [37, 43-45].
