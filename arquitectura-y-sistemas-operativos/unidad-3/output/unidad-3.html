Resumen General

Síntesis de los contenidos de la unidad según NotebookLM

Los SO modernos separan dos características que antes estaban unidas en el proceso: propiedad de recursos (espacio de memoria, PCB) y planificación/ejecución. La segunda se delega al hilo (thread), la unidad mínima de ejecución, también llamada "proceso liviano".

En un proceso multihilo, todos los hilos comparten el espacio de memoria, variables globales y archivos abiertos, pero cada hilo tiene su propio PC, pila y registros. Sin mecanismo de protección entre ellos, son 10–100 veces más rápidos de crear que un proceso.

Existen dos modelos de implementación: ULT (el Kernel no sabe que existen; rápido pero un hilo bloqueante paraliza todo el proceso) y KLT (el Kernel los gestiona; si un hilo se bloquea, el proceso continúa, pero cada cambio requiere un costoso doble cambio de modo).

Conceptos Clave

Términos esenciales extraídos del material

Hilo (Thread)
Unidad de planificación/ejecución dentro de un proceso. "Proceso liviano". Tiene su propio PC, pila y registros, pero comparte el espacio de memoria del proceso.
Multihilo
Capacidad del SO de soportar múltiples hilos ejecutándose simultáneamente dentro de un único proceso.
Memoria compartida entre hilos
Todos los hilos de un proceso comparten: espacio de memoria, variables globales, punteros a archivos, señales e info estadística.
Privado de cada hilo
Código propio, Program Counter (PC), registros de ejecución, pila (stack) y variables locales. Cada hilo tiene su propio contexto de ejecución.
ULT — User Level Thread
Hilos gestionados por una biblioteca en espacio de usuario. El Kernel no sabe que existen; solo ve el proceso completo.
KLT — Kernel Level Thread
Hilos conocidos y gestionados por el Kernel. Requiere syscall para cada operación de hilo, pero permite paralelismo real en multiprocesadores.
Condición de carrera
Problema de sincronización donde el resultado depende del orden de acceso de los hilos a la memoria compartida. Puede producir datos corruptos.
Deadlock (Interbloqueo)
Situación donde dos o más hilos se bloquean mutuamente esperando recursos que el otro tiene. Ninguno puede continuar.
Biblioteca de hilos
API para crear y gestionar hilos. Ejemplos: POSIX Pthreads, Win32 threads, Java threads.
Doble cambio de modo
Overhead de los KLT: cada cambio de hilo requiere saltar de Usuario a Kernel y volver, lo cual es más costoso que el cambio de hilo en ULT.

Desarrollo

Explicación detallada con ejemplos del material

Proceso vs. Hilo — La separación de responsabilidades

En los SO tradicionales, el proceso era tanto el propietario de recursos como la unidad de ejecución. Los SO modernos separan estos roles:

Proceso (recursos)Hilo (ejecución)
Espacio de direccionesProgram Counter propio
Variables globalesStack propio
Archivos abiertosRegistros propios
Procesos hijosVariables locales propias
PCB del procesoPCB de hilo propio
Sin protección entre hilos: como comparten el mismo espacio de memoria, si un hilo modifica una variable global, todos los demás hilos del proceso lo ven inmediatamente. No hay aislamiento.

¿Por qué usar hilos? Ventajas y ejemplos del material

Crear, finalizar o cambiar un hilo es entre 10 y 100 veces más rápido que hacerlo con un proceso completo. La comunicación entre hilos no requiere intervención del SO.

Ejemplo 1 — Procesador de texto

Un hilo lee los comandos del usuario; otro guarda en disco en paralelo. En monohilo el programa "colgaría" durante el guardado, sin poder atender el teclado. Con hilos, ambas acciones ocurren simultáneamente.

Ejemplo 2 — Servidor Web

Se crea un hilo por cada request entrante. Si la página está en caché, ese hilo responde de inmediato. Si no está, ese hilo se bloquea buscando en disco, pero los demás hilos siguen atendiendo otras peticiones en paralelo.

Estados de los Hilos

  • Creación: al crear un proceso se crea automáticamente un hilo por defecto. Se pueden crear más hilos, cada uno con su propio contexto y pila.
  • Bloqueo: el hilo guarda sus registros y se pausa esperando un evento (ej: lectura de disco). Permite que el SO ejecute otro hilo.
  • Desbloqueo: cuando ocurre el evento, el hilo se mueve a la cola de "Listos".
  • Finalización: el hilo completa su ejecución; se liberan registros y datos.
Detalle crítico: los estados globales (Suspendido, Finalizado) aplican al nivel del proceso. Si el SO suspende o finaliza un proceso, todos sus hilos son suspendidos o finalizados obligatoriamente.

ULT vs KLT — Modelos de implementación

ULT — User Level Thread
Gestión: biblioteca en espacio de usuario
Kernel: no sabe que existen
✓ Sin doble cambio de modo
✓ Algoritmo de planificación personalizable
✓ Funciona en SO que no soporten hilos
✗ Syscall bloqueante paraliza TODO el proceso
✗ No aprovecha multiprocesamiento real
KLT — Kernel Level Thread
Gestión: el Kernel directamente
Kernel: conoce todos los hilos
✓ Hilo bloqueado no detiene al proceso
✓ Asigna hilos a múltiples procesadores
✓ SO puede usar hilos internamente
✗ Doble cambio de modo por cada operación
✗ Más lento que ULT para cambio de hilo

Concurrencia y Sincronización

La concurrencia (ejecución solapada de hilos) mejora enormemente el rendimiento, pero genera riesgos cuando los hilos acceden a la misma memoria:

Condición de carrera

El resultado depende del orden de acceso a memoria compartida entre hilos. Puede corromper datos silenciosamente.

Deadlock (Interbloqueo)

Dos o más hilos se bloquean mutuamente esperando recursos que el otro posee. Ninguno puede avanzar.

Ambos problemas se resuelven con técnicas de sincronización como mutexes, semáforos y monitores (contenido de unidades posteriores).
