Procesos

Tecnicatura Universitaria en Programación

📋Resumen General

Un proceso es la abstracción central de cualquier sistema operativo: un programa en ejecución con su propio contexto, código, datos, pila y heap. El SO gestiona cada proceso mediante el BCP (Bloque de Control de Proceso), que almacena PID, estado, prioridad y registros del procesador. Los procesos pasan por distintos estados: desde el modelo simple de 2 estados (Ejecutando / No ejecutando) hasta el modelo de 7 estados que incorpora swapping. El cambio de contexto ocurre ante interrupciones, excepciones o system calls, y el procesador opera en dos modos: usuario y núcleo.

🔑Conceptos Clave

Proceso
Abstracción de un programa en ejecución. Compuesto por: Contexto de ejecución, Código (Texto), Datos globales, Pila (Stack) y Heap. Es una entidad activa, a diferencia del programa que es pasivo.
BCP — Bloque de Control de Proceso (PCB)
Estructura de datos que el SO crea para gestionar cada proceso. Contiene: PID (identificador único), Estado, Prioridad, PC (contador de programa), Datos de contexto, información de E/S, punteros a memoria y datos de auditoría.
Dispatcher
Pequeño programa del SO que se encarga de intercambiar los procesos en ejecución. Gestiona las colas y decide qué proceso corre a continuación.
Pseudoparalelismo
Ilusión de paralelismo generada cuando el CPU alterna entre procesos tan rápidamente (decenas o cientos de milisegundos) que el usuario percibe ejecución simultánea, aunque en realidad solo uno corre a la vez.
Modelo de 2 estados
El modelo más simple: un proceso está Ejecutando o No ejecutando. Los procesos esperan en una cola FIFO y el procesador los atiende con estrategia round-robin.
Modelo de 5 estados
Extiende el modelo de 2 estados. Los estados son: Nuevo (creado, sin cargar en RAM), Listo (en cola, listo para ejecutar), Ejecutando, Bloqueado (espera evento/E/S) y Saliente (terminó o fue abortado).
Swapping
Operación de mover una parte o todo el proceso de la memoria principal al disco. Se activa cuando todos los procesos están bloqueados y el procesador estaría ocioso. Es una operación de E/S.
Cambio de contexto (Context Switch)
Ocurre cada vez que el SO obtiene el control sobre el proceso en ejecución. Causas: interrupción (de reloj, de E/S), excepción (abort, fault, trap) o system call. Implica salvar y restaurar el contexto del proceso.
Modos de ejecución: Usuario y Núcleo
El procesador tiene 2 modos. Modo Usuario: solo puede hacer syscalls, referencias de memoria relativas al registro base. Modo Núcleo (Kernel): control absoluto, acceso a cualquier dirección física. El modo se almacena en el PSW.
Excepciones: Abort, Fault y Trap
Abort: error grave e irrecuperable (no se puede reiniciar). Fault: recuperable, reinicia la instrucción que falló (ej: page fault). Trap: continúa después de la instrucción que lo generó (ej: breakpoint, división por cero).
Modos de ejecución del SO
Núcleo sin procesos: SO fuera de todo proceso de usuario. Núcleo en procesos de usuario: SO en el contexto del proceso (evita doble cambio de contexto). Basado en procesos: SO como colección de procesos independientes (más modular).
Heap vs Pila (Stack)
Pila: datos temporales, llamadas a funciones, parámetros y puntos de retorno. Heap: asignación dinámica de memoria (estructuras dinámicas como listas, colas, pilas en memoria).

📖Explicación Detallada

🧩 Proceso vs Programa

Un programa es una entidad pasiva: un archivo en disco con instrucciones. Un proceso es ese mismo programa cuando está en ejecución, con todo lo necesario para correr: su contexto (registros del procesador), su código cargado en RAM, sus datos, su pila de llamadas y el heap para memoria dinámica.

Ejemplo: si tenés el archivo mi_app.exe en el disco, eso es un programa. Al hacer doble click, el SO crea un proceso: le asigna un PID, reserva memoria y crea el BCP.

📊 Modelos de estado

Modelo de 2 estados — El más simple:

Ejecutando ⇄ No Ejecutando

Modelo de 5 estados — Separa "No Ejecutando" en Listo y Bloqueado:

Nuevo → Listo ⇄ Ejecutando → Saliente
↕
Bloqueado

Transición clave: Ejecutando → Listo ocurre cuando el proceso agotó su tiempo máximo de CPU (time slice). Ejecutando → Bloqueado ocurre cuando necesita esperar E/S.

Modelo de 7 estados — Agrega swapping (procesos en disco): Listo/Suspendido y Bloqueado/Suspendido. Se activa cuando no hay procesos en estado Listo en memoria.

🔄 Cambio de contexto

Cuando el SO toma control, guarda el estado completo del proceso actual en su BCP y carga el estado del próximo proceso desde su BCP. Este mecanismo permite la multiprogramación.

  • Interrupción de reloj: el proceso agotó su tiempo → pasa a Listo.
  • Interrupción de E/S: un dispositivo terminó → procesos bloqueados pasan a Listo.
  • System call: el proceso pide un servicio al SO (abrir archivo, crear proceso).
  • Excepción: condición de error en el proceso (division por cero, page fault).

⚠️ Tipos de Excepciones

ABORT
Error grave e irrecuperable. El proceso termina en estado Saliente. Ej: errores de hardware, valores ilegales en tablas del sistema.
FAULT
Recuperable. Reinicia la instrucción que falló. Ej: page fault → SO carga la página, proceso pasa de Bloqueado a Listo y reintenta.
TRAP
Continúa después de la instrucción. Ej: breakpoint para depuración, división por cero (el SO decide si aborta o continúa).

⚙️ Modos de ejecución del SO

  • Núcleo sin procesos: el SO es una única pieza separada de todo proceso de usuario. Cada interrupción implica un cambio de contexto completo. Analogía: kernel monolítico.
  • Núcleo en procesos de usuario: el SO corre dentro del contexto del proceso de usuario. Ante una interrupción, solo cambia de modo (usuario → kernel) sin cambio de proceso completo. Ventaja: evita el doble cambio de contexto. Analogía: microkernel.
  • Basado en procesos: el SO es una colección de procesos independientes. Mayor modularidad y posibilidad de paralelismo. Analogía: diseño cliente-servidor.
