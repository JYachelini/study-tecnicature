Resumen General

Síntesis de los contenidos de la unidad según NotebookLM

Un proceso es un programa en ejecución — entidad activa, a diferencia del programa que es una entidad pasiva. En memoria ocupa cinco secciones: Contexto (registros), Texto (código), Datos, Stack y Heap.

El SO administra cada proceso a través del Bloque de Control de Proceso (BCP/PCB), que guarda su PID, estado, prioridad y contexto de registros. Los modelos de estados evolucionan: 2 estados (básico), 5 estados (Nuevo, Listo, Ejecutando, Bloqueado, Saliente) y 7 estados con Swapping (agrega Listo/Suspendido y Bloqueado/Suspendido).

El cambio de contexto guarda el estado del proceso actual y restaura el de otro. Es desencadenado por interrupciones, syscalls o excepciones (Abort, Fault, Trap). Todo ocurre bajo dos modos del procesador: Usuario (restringido) y Kernel (control total).

Conceptos Clave

Términos esenciales extraídos del material

Proceso
Programa en ejecución. Entidad activa con contexto, recursos y estado. Un programa es una entidad pasiva (archivo en disco).
Pseudoparalelismo
Ilusión de ejecución simultánea en un monoprocesador. El CPU alterna entre procesos tan rápido que parece paralelo. El verdadero paralelismo requiere múltiples CPUs físicas.
BCP / PCB
Bloque de Control de Proceso: estructura de datos con PID, estado, prioridad, contexto de registros y peticiones de E/S pendientes. Una entrada por proceso en la Tabla de Procesos.
Dispatcher (Despachador)
Pequeño programa del SO que gestiona la cola de procesos listos y asigna el procesador usando Round-Robin (FIFO).
Swapping
Técnica que mueve un proceso (parcial o totalmente) de RAM al disco para liberar memoria. Considerada una operación de E/S. Agrega los estados Suspendido al modelo.
Cambio de Contexto
El SO interrumpe el proceso actual, guarda su contexto en su BCP y restaura el contexto de otro proceso para ejecutarlo.
Cambio de Modo
Pasar de Modo Usuario a Modo Kernel (o viceversa). No siempre implica cambio de contexto: el SO puede atender la interrupción y retomar el mismo proceso.
Excepción — Abort
Error fatal e irreversible (hardware defectuoso, tablas corruptas). El proceso pasa directamente al estado Saliente.
Excepción — Fault
Error corregible. Ej: Page Fault. El proceso se bloquea, el SO corrige el problema y reinicia la instrucción que falló.
Excepción — Trap
Notificación después de ejecutar la instrucción; permite continuar sin reiniciar. Ej: breakpoints de debug, división por cero.
Stack (Pila)
Sección de memoria del proceso con variables locales, parámetros de funciones y direcciones de retorno. Crece hacia abajo dinámicamente.
Heap (Montículo)
Sección para asignación dinámica de memoria en tiempo de ejecución (listas, colas). Crece hacia arriba.

Datos
Heap ↑
· · ·

  • Contexto: registros PC, IR, PSW usados por el SO para gestionar el proceso.
  • Texto: instrucciones del programa (solo lectura).
  • Datos: variables globales y estáticas.
  • Heap: memoria dinámica (malloc/new). Crece hacia arriba.
  • Stack: variables locales, parámetros y retornos. Crece hacia abajo.

El SO mantiene cuatro tipos de tablas: Tablas de Memoria, de Dispositivos (E/S), de Archivos y de Procesos (donde está el BCP de cada proceso).

El BCP y sus campos clave

BCP/PCB es la estructura de datos fundamental para la multiprogramación. Sin él, el SO no podría pausar un proceso y retomarlo exactamente donde estaba.
  • PID: identificador único del proceso.
  • Estado: Nuevo / Listo / Ejecutando / Bloqueado / Saliente.
  • Prioridad: para scheduling.
  • Datos de contexto: valores actuales de PC, IR y otros registros — se guardan al pausar y restauran al reanudar.
  • Info adicional: peticiones de E/S pendientes, punteros a código/datos, tiempo de CPU usado.

Modelos de Estados

Modelo de 2 Estados

Ejecutando No Ejecutando

Los procesos no activos esperan en una cola FIFO gestionada por el dispatcher usando Round-Robin. Simple pero ineficiente: mezcla procesos listos con procesos bloqueados esperando E/S.

Modelo de 5 Estados

Nuevo Listo Ejecutando Bloqueado Saliente

Nuevo: proceso creado (tiene BCP y PID) pero aún no cargado en RAM. Listo: en memoria, esperando CPU. Bloqueado: esperando un evento de E/S. Saliente: finalizó o fue abortado.

Transiciones clave: Ejecutando→Listo (agotó tiempo o prioridad), Bloqueado→Listo (llegó el evento esperado), Ejecutando/Listo→Saliente (si el proceso padre lo finaliza).

Modelo de 7 Estados — Con Swapping

Nuevo Listo Ejecutando Bloqueado Saliente Listo/Susp. Bloqueado/Susp.

Si la RAM se llena de procesos Bloqueados, el CPU queda ocioso. El SO resuelve esto con Swapping: mueve procesos al disco. Listo/Suspendido: en disco, listo para ejecutarse si vuelve a RAM. Bloqueado/Suspendido: en disco y además esperando un evento.

Cambio de Contexto y Modos de Ejecución

Modo Usuario

Restringido. La memoria se accede de forma relativa. Solo puede pedir servicios al SO mediante Syscalls.

Modo Kernel

Control total. Referencias físicas a memoria. Aquí se ejecutan scheduling, swapping, creación de procesos e interrupciones.

El Cambio de Contexto ocurre cuando el SO toma el control, salva el contexto del proceso actual en su BCP y carga el contexto del siguiente proceso. Lo desencadenan:

INT
Interrupciones: eventos asíncronos del hardware (reloj que agota el tiempo, dispositivo de E/S que terminó).
SYS
Syscalls: petición voluntaria del proceso (crear archivo, leer disco, crear proceso hijo).
EXC
Excepciones: errores síncronos internos del proceso (ver debajo).
Importante: un Cambio de Modo (Usuario→Kernel) no siempre implica un Cambio de Contexto. Si la rutina del SO termina rápido y retoma el mismo proceso, solo hubo cambio de modo.

Tipos de Excepciones

Abort — Fatal

Error irreversible (hardware defectuoso, tablas corruptas). El proceso pasa directo a estado Saliente. No hay recuperación.

Fault — Corregible

Error que el SO puede resolver. Ej: Page Fault. El proceso se bloquea, el SO trae la página a RAM, y la instrucción se reinicia.

Trap — Notificación

Se notifica justo después de la instrucción. El proceso continúa (no reinicia). Ej: breakpoints, división por cero para debug.

Cómo se Ejecuta el propio SO

1
Núcleo sin procesos: el SO corre totalmente aislado como software monolítico. Cada interrupción genera un cambio de contexto completo.
2
Dentro de procesos de usuario: las rutinas del SO se ejecutan dentro del entorno (y pila) del proceso actual. Evita dobles cambios de contexto — muy eficiente.
3
Basado en procesos: el SO es una colección de procesos independientes en modo kernel. Máxima modularidad y permite paralelismo real entre servicios del sistema.
